# Ream Meeting #78 Recap

## Overview

This week's sync focused on devnet stability, hardening the Hive test harness, and
the first steps of execution-layer integration. The most stable
cross-client devnet of the post-quantum effort to date kept finalizing across tens
of thousands of slots, while contributors tracked down a Ream race condition and a
proof-aggregation performance issue that had been blocking the Hive sync tests.
Execution-layer/consensus integration kicked off with a multi-phase plan, spec
compatibility work continued on epoch processing and data availability sampling,
and several new EPF participants introduced themselves and started contributing to
Hive.

## Development Progress

### Devnet Stability & Scaling

- **Cross-client devnet**: [Shariq] Ran ethlambda ↔ ream devnets, including a
  6-node devnet that finalized to around 55,000 slots and a 24-validator devnet
  (12 ethlambda + 12 ream) that is still finalizing past 90,000 slots — the most
  stable devnet of the whole PQ effort so far, though limited to just ream and
  ethlambda. zeam can't run on multiple subnets yet because of their new networking
  library, so their client is on hold; Lantern and possibly Grandine will be added
  once they're ready.
- **Multi-machine setup**: [Shariq] The cross-client run uses four aggregator
  machines (16 cores, 32 GB RAM each) with two validators on each 8-core / 16 GB
  machine.
- **Local devnets**: [Kayden] Running ream at a smaller scale locally with 4 subnets
  on lean-quickstart, with a PR updating the build-blocks function to follow.
- **Aggregation & finality tuning**: [Unnawut] Used lean-start to spin up a devnet
  on the aggregation side (multi-subnet was blocked last week) and started measuring
  the finalized-slot delay to find the maximum validator and subnet counts a single
  instance can sustain — 4-core machines for regular validators, 16-core for
  aggregators — aiming for a configuration that holds three-slot finality.

### Hive Test Harness

- **Merges & cleanup**: [Derek] Landed a fair amount for Hive and expects it to be
  in good shape once everything merges — finished the devnet-5 asset tests, adjusted
  the sim limit on the Hive deploy, and reworked the sync-suite timeout wrappers.
- **Removing retries**: [Derek] Together with Kolby, removed the retry logic from
  Hive; while useful early on for stability, retries were masking transient client
  crashes. The only remaining restart path is when a leanSpec helper (not the client)
  has an issue during a test, which is a candidate for a small EPF stability project
  on the leanSpec client itself.
- **Panic fix**: [Derek] Found and fixed a condition in the RPC-compat shared client
  test that could panic and end an entire suite early.
- **Sync suite fix**: [Sahil] Of six sync tests, five were failing on the helper-mesh
  timeout (resolved by Derek's timeout PR). The remaining failure was ream-specific:
  with the helper mesh unused, proof aggregation was too slow and the finalized slot
  stayed at zero — fixed by adding native CPU flags (AVX2) for Linux x86 in ream's
  config.toml, which dropped proof aggregation from 3–5 seconds to 20–30 milliseconds
  and moved finalization from slot 0 to slot 6.
- **New Hive contributors**: [Jeff] and [Richard], both new EPF participants, are
  ramping up on Hive — understanding the simulator and architecture, running
  simulations, and (for Richard) looking at improving the cross-client transition
  tests by comparing results across clients. Both will sync with Derek on direction.

### Ream Client Fixes

- **redb race condition**: [Derek] While trying out lean-start (praised for its
  auto-restart on client crashes), stumbled on a rare ream race condition: in
  backfill and prune-stale-pending-blocks, a couple of functions returned an open
  transaction on a redb table that, if held long enough, could panic and crash ream.
  Patched so those functions return only the needed data and no longer hold an open
  transaction; PR expected the same day.
- **Shared fixes image**: [Shariq] Has a ream image carrying many accumulated fixes
  and will share it; hasn't hit Derek's race condition and will look out for the PR.
- **Test cleanup**: [Sahil] Removing unit tests in ream that are already covered by
  the fixtures, and adding support so the fork-choice fixtures can also run for
  devnet-5. Also resolved a dependency version conflict between ream and reth that
  surfaced when the two were built together.

### Execution-Layer Integration

- **EL/CL integration**: [Soham] Wrote the plan for the execution-layer integration
  in ream and, with Sahil and Sameer, opened issues on the ream GitHub and started
  phase one, aiming to finish it within the next few days.
- **Phase breakdown**: [Sahil] The integration has three phases — phase one wires
  fork choice into ream-side communication (expected done by end of week), with the
  engine-API integration coming in phase two.
- **Execution engine**: [Tosin] Started looking into the execution engine and its
  resource usage; expects related commits to start appearing within a day or two.

### Spec Compatibility & Data Availability Sampling
