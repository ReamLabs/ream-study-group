# Ream Meeting #71 Recap

## Overview

This meeting covered ongoing progress across benchmarking, CI infrastructure, and Hive testing. The team discussed updates to leanBench, including its migration to the Lean Ethereum codebase and a performance reassessment after switching benchmarks to the devnet-4 branch, which uses a formally verified but slower XMSS implementation. The team also reviewed plans for setting up public dashboards and CI pipelines, as well as ongoing work on Hive visualization improvements and sync tests, where memory issues with the leanSpec client have been the main blocker.

## Development Progress

### CI & Dashboards

- **CI pipeline**: A CI workflow will also be configured today, pending access being provided (Shariq, with support from Kolby).
- **Hardware exploration**: Brief discussion on using M5 Mini for raw CI jobs, with isolation concerns to be revisited later (Kolby, Shariq).

### Benchmarking (leanBench)

- **Codebase migration**: leanBench has been moved over to the Lean Ethereum organization and the code has been cleaned up significantly (Unnawut).
- **Benchmark target switch**: Benchmarks were switched from main to the devnet-4 branch as suggested by Thomas, resulting in a ~3x performance drop due to the formally verified XMSS used in the devnet-4 and devnet-5 branches of leanMultisig (Unnawut).
- **Topology adjustment**: The previous goal of ~1000 validators might need to be revised down to ~300 validators per slot, which still aligns with devnet-4's current scale (Unnawut).

### Hive Testing

- **Visualization improvements**: The Lean latest page has been promoted to the main page of Hive, and a client total grid plus a missing diagonal (client-against-itself) cell in the interop grid have been added (Derek).
- **Sync tests**: Two new sync tests have been added to Hive, inspired by an earlier Ream issue where a client could fall behind head without being able to catch up. The tests themselves are working as intended (Derek).
- **leanSpec client memory issues**: The leanSpec reference client is hitting memory issues during sync tests. Current workaround is a constrained topology (two leanSpec clients, one with two validators and one with a single validator), but further diagnosis inside the leanSpec client itself is likely needed (Derek).
